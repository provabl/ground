Customizing ground

All ground behaviour is driven by ground.yaml. Start with the example, adjust for your institution, and use ground deploy --dry-run to preview before applying.

ground.yaml full reference

Copy ground.example.yaml and fill in the required fields. Everything else has sensible defaults.

ground.yaml — full reference with defaults 
# REQUIRED ─────────────────────────────────────────────────────────
org:
  name: "My Research University"          # human name for SSP/reports
  region: us-east-1                       # primary deployment region
  management_account_id: "123456789012"   # management account (required)
  audit_email: audit@example.edu          # Log Archive account email
  logging_email: logging@example.edu      # optional: separate logging account
  workload_ous:                           # custom sub-OUs to create
    - research
    - sandbox

# NETWORKING (Phase 2) ─────────────────────────────────────────────
network:
  transit_gateway: true                   # deploy TGW in Network Hub account
  cidr_block: "10.0.0.0/8"               # supernet for all VPCs
  vpc_endpoints:                          # services to endpoint-ize
    - s3
    - ec2
    - sts
    - ssm
    - secretsmanager
    - kms
    - logs

# IDENTITY ─────────────────────────────────────────────────────────
identity:
  identity_center: true                   # deploy IAM Identity Center
  instance_arn: ""                        # set if Identity Center already exists

# LOGGING ──────────────────────────────────────────────────────────
logging:
  retention_days: 365                     # audit log retention (S3 lifecycle)
  bucket_name: ""                         # auto-generated if empty

# SECURITY ─────────────────────────────────────────────────────────
security:
  guardduty: true                         # enable GuardDuty org-wide
  security_hub: true                      # enable Security Hub + NIST 800-53
  macie: true                             # enable Macie org-wide

# TAGGING ──────────────────────────────────────────────────────────
tagging:
  required_tags:                          # per-tag OR logic SCPs
    - project
    - environment
    - owner
    - data-classification

Deploying individual stacks

ground deploy runs all four stacks in sequence. If you only need certain layers — or want to deploy incrementally — you can target stacks individually using the AWS CLI or your IaC output.

# Deploy only the logging foundation (minimum for attest to work)
ground deploy --dry-run # shows all 4 stacks

# Generate Terraform or CDK and apply selectively
ground deploy --output terraform
cd ground-terraform

# Apply only the logging and security resources
terraform apply -target=aws_cloudformation_stack.ground_logging \
                -target=aws_cloudformation_stack.ground_security

Minimum viable (for attest)

Deploy only logging and security stacks. This gives attest what it needs to run attest scan and assess posture — CloudTrail, Config, GuardDuty, Security Hub.

ground deploy
# then skip accounts + identity if not ready

IaC output modes

ground generates CloudFormation (default), Terraform HCL, or CDK TypeScript. The CloudFormation path deploys directly; Terraform and CDK generate artifacts for you to apply with your existing toolchain.

ground deploy --output cloudformation  # default
ground deploy --output terraform        # → ground-terraform/
ground deploy --output cdk             # → ground-cdk/

Disabling services

Set any security service to false to skip it. Be aware of the compliance implications — disabling GuardDuty, for instance, means attest will report that control as a gap during attest scan.

# ground.yaml — disable Macie if not needed for your data types
security:
  guardduty: true
  security_hub: true
  macie: false     # skip Macie — attest will flag nist-800-171-r2 3.14.x as partial

Compliance impact of disabling services

  • Disabling GuardDuty — gap against NIST 800-171 §3.14.7, CMMC SI.L2-3.14.7, HIPAA §164.308(a)(1)
  • Disabling Security Hub — attest loses posture data; scan results will be incomplete
  • Disabling Macie — gap against HIPAA minimum necessary and some NIH GDS data discovery requirements

Custom workload OUs

The five default OUs (Security, Infrastructure, Research, Sensitive Research, DoD/CMMC) are always created. Add institution-specific OUs under workload_ous.

# ground.yaml — add institution-specific OUs
org:
  workload_ous:
    - research          # → /Research/ OU
    - sandbox           # → /Research/Sandbox/ sub-OU
    - hpc-cluster       # → /SensitiveResearch/HPC/ sub-OU

Custom OUs inherit the SCPs of their parent tier. They are tagged with ground:tier matching their parent, which attest uses for SSP system boundary auto-discovery.

Custom required tags

The default tagging SCP enforces five attest:* tags. Add institutional tags to the list — each becomes its own Deny statement (OR logic, not AND).

# ground.yaml — add institutional tags
tagging:
  required_tags:
    - project
    - environment
    - owner
    - data-classification
    - cost-center        # institution-specific
    - grant-number       # for grant reporting
